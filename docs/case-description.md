# FeedbackCraft — 智能绩效反馈生成器

## 一、问题诊断

### 1.1 现状分析

绩效反馈是管理者最频繁的 HR 工作之一，但长期面临三个核心痛点：

**时间成本高。** 根据行业调研，管理者平均需要 30-60 分钟撰写一份结构完整、内容具体的绩效反馈。在季度评估季，一位管理 5 人团队的经理仅写反馈就需要半天时间，严重影响核心业务工作。

**质量参差不齐。** 不同管理者的反馈质量差异巨大。经验丰富的管理者能写出有深度的反馈，但多数管理者的输出停留在"工作态度积极""团队协作好"等模糊套话，缺乏具体事例和可执行的改进建议。这让员工感到反馈不专业、不公平。

**AI 工具的局限。** 虽然已有通用 AI 工具可以辅助写作，但它们缺乏绩效管理的领域知识——不知道反馈应包含哪些结构、如何平衡正负评价、如何写出 SMART 标准的发展建议。直接使用通用 AI 生成的反馈往往结构混乱、用词不当。

### 1.2 问题本质

核心问题不是"不会写"，而是"没有标准化的流程和专业框架"。管理者需要的不是一个聊天机器人，而是一个**懂绩效管理的专业助手**，能够引导他们收集关键信息，并按照 HR 最佳实践生成结构化反馈。

## 二、方案设计

### 2.1 核心创新

我们的核心创新是：**将绩效反馈的专业流程封装为一个 AI Agent Skill**，而非构建一个独立的 Web 应用。

**为什么选择 Skill 而不是独立应用？**

| 维度 | 独立 Web 应用 | AI Agent Skill |
|------|-------------|----------------|
| 部署 | 需要服务器、域名、维护 | 单个 Markdown 文件，复制即用 |
| 数据安全 | 数据上传到第三方服务器 | 数据全程本地处理 |
| 使用门槛 | 需要打开新应用、注册账号 | 在现有 AI 助手中直接调用 |
| 可扩展性 | 需要开发新功能 | 修改配置文件即可适配 |
| 成本 | 需要服务器费用 | 零成本 |

**设计哲学：** 我们相信，最好的工具是"无处不在"的。通过 Skill 格式，FeedbackCraft 可以嵌入到开发者日常使用的 AI 编程助手中，无需额外学习成本，打开对话即可使用。

### 2.2 架构设计

```
用户输入 → Skill 引导采集 → 专业框架处理 → 结构化输出
    ↓              ↓              ↓              ↓
 员工信息      逐步询问        5部分模板       可直接使用
 成就/改进     补充细节        质量规则        的反馈文本
 语气偏好      引导具体化      行业适配
```

**Skill 架构包含 8 个核心模块：**

1. **Overview** — 快速理解 Skill 用途
2. **When to Use** — 触发场景指引
3. **Input Collection** — 结构化信息采集
4. **Output Structure** — 5 部分输出模板
5. **Tone Guidelines** — 3 种语气风格
6. **中文输出指南** — 中文场景用词规范
7. **行业适配指南** — 技术/销售/产品/职能 4 个行业模板
8. **负面反馈话术指南** — 敏感话题处理框架

### 2.3 质量保障机制

内置 7 条质量规则，确保输出专业性：

1. 每句反馈必须引用具体事例
2. 禁止使用模糊措辞
3. 正负评价保持 2:1 比例
4. 发展建议必须符合 SMART 原则
5. 绝不编造数据或成就
6. 信息不足时主动询问
7. 尊重隐私，建设性表达

## 三、AI 工具选型

### 3.1 为什么用 Claude Code / Codex

选择 Claude Code 和 Codex 作为主要运行平台，理由如下：

**复用现有生态。** 管理者日常已在使用 AI 编程助手，Skill 可以直接嵌入其工作流，无需打开新应用。这比独立 Web 应用的使用门槛更低。

**隐私保护。** 企业绩效数据高度敏感。Skill 在本地 AI 助手中处理，数据不经过任何第三方服务器，满足企业数据安全要求。

**可扩展性。** 基于通用 Skill 格式，任何开发者都可以 fork 并适配自己的绩效管理框架、企业文化和评估标准。

### 3.2 为什么不选其他方案

| 方案 | 不选的原因 |
|------|-----------|
| Coze 智能体 | 第三方平台，在腾讯比赛中使用竞品产品不合适 |
| 独立 Web 应用 | 需要服务器、用户账号、数据存储，增加复杂度和风险 |
| 前端直调 API | API Key 暴露在前端，有安全风险 |

## 四、迭代记录

### v1.0 — 基础模板
最初版本仅包含输出模板（5 部分结构），AI 按模板填充内容。测试发现：缺乏输入引导，用户提供的信息不够具体，导致输出仍然泛泛而谈。

### v1.1 — 结构化输入
增加了输入采集模块，AI 主动询问必填信息，并提供"好例子 vs 坏例子"对比引导。测试发现：输出质量提升，但不同场景的语气不匹配。

### v1.2 — 语气系统
增加了三种语气风格（正式/温和/直接），每种风格有独立的措辞指南和示例开头。测试发现：AI 偶尔会编造数据或使用模糊措辞。

### v1.3 — 质量规则和反模式
增加了 7 条质量规则和常见错误表，明确禁止编造数据、模糊措辞等行为。加入完整输入→输出示例作为参考模式。

### v2.0 — 中文优化与行业适配（当前版本）
新增中文输出指南（用词规范、句式模板、标点规范）、4 个行业适配模板（技术/销售/产品/职能）、负面反馈话术指南。

## 五、效果评估

### 5.1 量化指标

| 指标 | 传统方式 | 使用 FeedbackCraft | 提升 |
|------|---------|-------------------|------|
| 撰写时间 | 30-60 分钟 | 3-5 分钟 | 10x |
| 结构完整性 | 随机 | 5 部分标准结构 | 100% |
| 语言专业性 | 参差不齐 | 统一专业框架 | 显著提升 |
| 可执行性 | 泛泛而谈 | SMART 建议 | 显著提升 |

### 5.2 质性评估

**优势：**
- 流程标准化，消除管理者之间的质量差异
- 内置 HR 最佳实践，提升反馈专业性
- 零成本、零部署、隐私安全
- 可复用、可扩展、可定制

**局限性：**
- 当前版本依赖用户提供具体事例，如果用户无法回忆关键事件，输出质量会下降
- 未来可考虑接入项目管理工具（如 Jira、飞书）自动提取关键事件

### 5.3 未来方向

- 支持更多反馈场景（360 度反馈、项目复盘、入职评估）
- 接入企业绩效系统实现一键导入导出
- 增加多语言支持
- 开发可视化仪表盘版本
